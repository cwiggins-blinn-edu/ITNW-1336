The Root User

Click to flip

The single identity created on initial signup. Holds absolute, unrestrictive billing and deployment capabilities. **Critical SAA-C03 best practice**: Delete root API keys, enable MFA immediately, and create standard IAM users.

Flashcard 2

IAM User vs. Role

Click to flip

An **IAM User** represents a persistent entity with long-term secrets (passwords, fixed access keys). An **IAM Role** has no persistent credentials; trusted operators dynamically assume roles to acquire short-lived keys through AWS STS.

Answer
Flashcard 3

IAM Group

Click to flip

A simple organizational collection of IAM users. **Core Exam Trap**: A group is not a true identity principal. Therefore, you *cannot* use an IAM Group inside the `"Principal"` field of S3 buckets or KMS key trust policies.

Answer
Flashcard 4

JSON Policy Elements

Click to flip

JSON policy files declare elements: **Effect** (Allow/Deny), **Action** (programmatic APIs targeted), **Resource** (ARNs mapped), and optional **Condition** qualifiers (restricting operations by source IP, dates, or MFA).

Answer
Flashcard 5

Principle of Least Privilege

Click to flip

A critical design pillar declaring that users and apps should only possess the minimum required credentials to perform specific tasks. Avoid generic administrators (`*:*`) and assign context-managed resources.

Answer

Main Lesson

Core Educational Blueprint

Important Certification Concept: Evaluation Logic Flow

AWS security starts with a **Default Deny** posture. If an evaluation encounters a single explicit **Deny** across *any* applicable policy (Identity, S3 Resource, or Organization SCP), the access request is instantly rejected, overriding any other explicit **Allow** statements.

Anatomy of an IAM JSON Policy Document

Take a look at a typical read-only JSON policy document for an S3 Coffee Recipes storage container:

JSON Syntax 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSpecificS3BucketRead",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::awscafe-recipes",
        "arn:aws:s3:::awscafe-recipes/*"
      ],
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
Version Control

Always use **"2012-10-17"**. If omitted, AWS treats variables and complex conditions under legacy structures which will fail verification.

Resource Format (ARN)

Amazon Resource Names format: `arn:partition:service:region:account:resource-id`. Note that S3 lacks region indicators as it is a global namespace.

Case Study

The AWS Café Scaling Journey

As Frank and Martha scaled their small business from static coffee catalogs to full ecommerce points-of-sale, they migrated from critical early configuration failures to solid, scalable cloud security:

Phase 1: Startup Disaster
  • **Daily Root User Logins**: Frank used absolute credentials on local shared PCs.
  • **Excessive Permissions**: Accountants possessed keys allowing modification of live databases.
  • **Exposed long-term keys**: Application code committed directly to github contained explicit credentials.
Phase 2: Scale & Secure
  • **Granular IAM Users**: Enforced individual credentials & mandatory Multi-Factor Authentication (MFA).
  • **Least Privilege Groups**: Segregated POS users from back-end infrastructure logs.
  • **IAM Roles for Web App**: Enabled zero administrative password storage on virtual machines.
